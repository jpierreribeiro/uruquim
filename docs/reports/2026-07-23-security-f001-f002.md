# Security report F-001 + F-002 — uruquim web stack

Date: 2026-07-23
Scope: authorized testing of the uruquim web framework (`web/`) and its
vendored transport backend (`vendor/odin-http`). Method: source review,
raw-socket fuzzing, race harness, AddressSanitizer instrumentation, gdb
triage. Lab artifacts: `/tmp/attacklab/` (target app, fuzzers, PoCs, logs).

Two findings. F-001 is in the vendored backend (report upstream). F-002 is in
uruquim's own transport adapter and is the more serious of the two.

---

## F-002 — Use-after-free in the WP8 transport adapter (URUQUIM CODE)

**Severity: HIGH. CWE-416 (Use After Free).**

### Affected code

`web/internal/transport/odin_http_adapter.odin`:

- `catch_all` (line 222): `exchange := new(Exchange, context.temp_allocator)`
- `dispatch_exchange` (lines 264-287): retains `exchange` across event-loop
  ticks via `nbio.next_tick_poly`
- vendored `server.odin` `clean_request_loop` (line 353):
  `free_all(context.temp_allocator)`

### Root cause

`Exchange` is allocated in the connection's per-request temp arena and holds
`req`/`res` pointers into `conn.loop`. When the synchronous handler lane is
busy, `dispatch_exchange` defers itself:

```odin
if !http.handler_lane_enter(res) {
    nbio.next_tick_poly(exchange, proc(_: ^nbio.Operation, exchange: ^Exchange) {
        dispatch_exchange(exchange)   // fires on a later tick
    })
    return
}
```

The pending tick callback keeps `exchange` alive across arbitrary event-loop
iterations, but nothing ties that lifetime to the connection. If the client
disconnects in that window, the backend close path runs
`connection_close -> clean_request_loop -> free_all(temp_arena)`, releasing
the whole arena — including the retained `Exchange`. When the queued tick
fires, `dispatch_exchange` dereferences the freed object at line 265
(`res := exchange.res`). `exchange.req`/`exchange.res` dangle the same way.

### Trigger (remote, unauthenticated)

1. Lane contention: one slow handler occupies the single synchronous Handler
   lane (lab: `GET /slow` sleeps 300 ms; any blocking app endpoint works).
2. A second request whose body arrives while the lane is busy, followed by an
   immediate TCP RST (`SO_LINGER {1,0}`), so the close path runs at once.

### Evidence

ASan (`/tmp/attacklab/uaf_rst.log`):

```
ERROR: AddressSanitizer: SEGV ... caused by a READ memory access
#0 transport::dispatch_exchange  odin_http_adapter.odin:265:2
#1 _proclit$anon-5               odin_http_adapter.odin:272:4   <- next_tick closure
#6 http::handler_lane_enter      server.odin:713
#7 transport::dispatch_exchange  odin_http_adapter.odin:270:2
#8 transport::on_body            odin_http_adapter.odin:260:2
```

gdb (`/tmp/attacklab/gdb_race2.log`): `exchange=0x7ffdeb296258`,
`Cannot access memory at address 0x7ffdeb296260` — freed object.

Reproducer: `/tmp/attacklab/poc_uaf.py` (reliable server kill).

### Exploitability

A true UAF, not a contained assert. The freed arena memory can be reclaimed
by later requests before the stale tick fires, and the stale path goes on to
`write_response(res, out)` / `http.respond(res)` (lines 285-286) — reads and
a response write through attacker-influenceable recycled pointers. Proven to
reliable remote crash; code execution not attempted.

### Fix applied (2026-07-23)

The `next_tick` retry was removed rather than made lifetime-safe. A deferred
dispatch can never run safely: everything it names (`req`/`res` into
`conn.loop`, the inbound views, the Exchange itself) lives in the connection
temp arena, and a client that disconnects before the retry — or even stays
connected while `clean_request_loop` tears the request down — leaves the
retry dereferencing freed memory. Refusal is the only sound answer. On a
failed `handler_lane_enter` the adapter now answers 503 and closes, while the
response state is still valid:

```odin
if !http.handler_lane_enter(res) {
    http.headers_set_close(&res.headers)
    res.status = http.Status.Service_Unavailable
    http.respond(res)
    return
}
```

Validated (ad-hoc, ASan+debug build): baseline traffic intact; 20 rounds of
the original trigger (slow handler holding the lane + victim request +
immediate RST) produce no crash and no ASan event. Previously this reliably
SEGV'd at `odin_http_adapter.odin:265`.

### Known remaining issue (out of scope, pre-existing)

Under a SUSTAINED RST flood the server stops accepting (all threads alive,
listen backlog fills, no crash) — a liveness wedge in the vendored
lane/accept machinery, present identically on the unfixed code (where it
manifests as the F-002 crash first). Not introduced by this fix; tracked as
follow-up hardening for the WP71 bounded-concurrency path.

### Earlier fix directions (superseded by the refusal above)

- Own the `Exchange` across the retry / refcount against `connection_close`.
- Cancellation-aware retry (remove pending `next_tick` on close).

## F-001 — Remote DoS via negative chunk-size (VENDORED odin-http)

**Severity: HIGH (DoS). CWE-20 / CWE-617. REPORT UPSTREAM (laytan/odin-http) —
local fix was reverted; this repo carries the bug via the vendor snapshot.**

### Affected code

`vendor/odin-http/body.odin` `_body_chunked.on_scan` (lines 229-252) ->
`vendor/odin-http/scanner.odin` `scan_num_bytes` (line 21 assert).

### Root cause

The chunk-size line is parsed with `strconv.parse_int(..., 16)`, which does
not range-check: oversized hex wraps mod 2^64 and returns `ok = true` with a
negative value (verified: `ffffffffffffffff` -> -1). The negative `size`
bypasses the `max_length` guard (`builder_len + size` underflows) and becomes
the scanner's expected byte count `split_data = rawptr(uintptr(size))`;
`scan_num_bytes` then hits `assert(n >= 0)` and the process aborts. One
unauthenticated request kills the server.

### Trigger

```
POST / HTTP/1.1
Host: x
Transfer-Encoding: chunked

ffffffffffffffff
A
```

Reproducer: `/tmp/attacklab/poc.py`. Contained to DoS by Odin's runtime
bounds checks even with `-disable-assert` (`data[:n]` with n=-1 traps); no
memory-corruption primitive obtained on stock builds.

### Fix (suggested for upstream)

Reject at the parse site: `if !ok || size < 0 { ... .Bad_Read_Count ... }`,
and compare `size > max_length - builder_len` instead of
`builder_len + size > max_length`. Applied and verified locally during
testing, then reverted per maintainer decision (report upstream instead).

## Surfaces reviewed with no finding

- `web/errors.odin` fixed `error_buffer`: bounded, `#assert`-proven.
- `web/request_id.odin` `request_id_buffer[64]`: charset+length validated.
- `web/static.odin` `static_etag_buffer[40]`: 35-byte worst case.
- `web/response.odin` `response_headers[12]`: worst case exactly 12.
- Adapter `copy_response` / keep-alive happy path: ASan clean over 50
  pipelined requests (the UAF needs lane contention + disconnect, not the
  happy path).
